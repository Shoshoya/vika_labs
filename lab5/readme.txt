# Lab Work 5: Network Security & Man-in-the-Middle (MitM)

## Мета
Зрозуміти обмеження захисту `HttpOnly` проти мережевих атак та навчитися використовувати прапорець `Secure` для захисту трафіку[cite: 3].

## Хід роботи та спостереження

### Task 1: Зловмисний проксі-сервер (The Malicious Proxy)
* **Що було зроблено:** Створено проксі-сервер на порту 8080, який перенаправляє запити на основний сервер (Port 3000)[cite: 3]. У режимі `Breach Mode` сервер перехоплює та виводить у термінал заголовок `Cookie` з усіх вхідних запитів[cite: 3].
* **Результат:** Незважаючи на те, що cookie має прапорець `HttpOnly`, проксі-сервер успішно перехопив `SessionID` та відобразив його в консолі[cite: 3].
* **Аналіз:** `HttpOnly` захищає cookie лише від доступу через JavaScript у середовищі браузера[cite: 3]. Однак він не захищає від зловмисника, який контролює мережевий шлях (шлюз або проксі), оскільки проксі бачить сирі HTTP-заголовки[cite: 3].

### Task 2: Посилення захисту (The Secure Flag)
* **Що було зроблено:** У маршрут `/login` на основному сервері додано атрибут `Secure` до заголовка `Set-Cookie`[cite: 3].
* **Результат:** При доступі через проксі (`http://localhost:8080`) браузер перестав надсилати cookie на сервер[cite: 3].
* **Висновок:** Прапорець `Secure` гарантує, що конфіденційні токени ідентифікації будуть передаватися лише через зашифровані з'єднання (HTTPS)[cite: 3]. Це захищає їх від перехоплення в мережі через відкритий текст (plain-text sniffing)[cite: 3]. Оскільки наше з'єднання з проксі було по протоколу HTTP, браузер відмовився передавати cookie з міркувань безпеки[cite: 3].
